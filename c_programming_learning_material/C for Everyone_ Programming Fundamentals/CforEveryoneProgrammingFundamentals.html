The source code written by programmers is first stored in a file, let the name be “program.c“. This file is then processed by preprocessors and an expanded source code file is generated named “program.i”. This expanded file is compiled by the compiler and an object code file is generated named “program.obj”. Finally, the linker links this object code file to the object code of the library functions to generate the executable file “program.exe”.

Preprocessor programs provide preprocessor directives that tell the compiler to preprocess the source code before compiling. All of these preprocessor directives begin with a ‘#’ (hash) symbol. The ‘#’ symbol indicates that whatever statement starts with a ‘#’ will go to the preprocessor program to get executed. We can place these preprocessor directives anywhere in our program.

  • The preprocessor is a program that runs before the compiler and performs some tasks, such as including header files, replacing macros, and removing comments.
  • can perform various tasks, such as including header files, defining macros, or performing conditional compilation. Some examples of preprocessor directives are #include, #define, #ifdef, and #endif.

Here variables & identifiers means different concept

Variables and identifiers are related but not the same concepts in C programming language. Here is a brief summary of the difference between them:

  • Variables are used to name a memory location, which holds a value that may be modified during program execution. In C, all the variables must be declared before use.
  • Therefore, all variables are identifiers, but not all identifiers are variables. For example, float area; here float is a keyword, and area is an identifier. The identifier area is a name given to a variable which will store a float value.
  • The term identifier just means "a name given to something" — it could be a variable, function, type, macro, or label. Only some of those identifiers refer to variables.

#include <stdio.h>

int main()

{

    int x = 10; // x is a variable and an identifier

    printf("Hello, world!\n"); // printf is a function and an identifier

    return 0;

}

In this code, x is a variable and an identifier. It is a variable because it names a memory location that stores an integer value, which can be changed during the program execution. It is also an identifier because it is a user-defined word that follows the naming rules of C.

On the other hand, printf is a function and an identifier. It is a function because it performs a specific task, which is printing a message to the standard output. It is also an identifier because it is a user-defined word that follows the naming rules of C.

However, printf is not a variable. It does not name a memory location that holds a value, and it cannot be modified during the program execution. Therefore, printf is an example of an identifier that is not a variable.

The calculation part in C: (here the data conversion happens in the background & since at the last I have used

An integer data type to show the output only 42 was showcased.

  1. First, the parentheses are evaluated from left to right. This means that the expression inside the parentheses is calculated before anything else. In this case, the expression is miles + yards / 1760.0.
  2. Next, the division is performed before the addition, because division has a higher precedence than addition in C. This means that the value of yards / 1760.0 is calculated first, and then added to the value of miles. Note that 1760.0 is a floating-point literal, which means it has a decimal point. This makes the division a floating-point division, which preserves the fractional part of the result. For example, if yards is 385, then yards / 1760.0 is 0.21875.
  3. After the division, the addition is performed. This means that the value of miles is added to the value of yards / 1760.0. For example, if miles is 26, then miles + yards / 1760.0 is 26.21875.
  4. Finally, the multiplication is performed. This means that the value of 1.609 is multiplied by the value of miles + yards / 1760.0. Note that 1.609 is also a floating-point literal, which makes the multiplication a floating-point multiplication, which preserves the fractional part of the result. For example, 1.609 * (miles + yards / 1760.0) is 42.195.
  5. However, since you have used the integer data type for the kilometres variable, it can only store whole numbers. Therefore, when you assign the value of 1.609 * (miles + yards / 1760.0) to it, it will truncate the decimal part and only keep the integer part. That’s why the output is 42 instead of 42.195.

To summarise, the calculation will be done in C as follows:

1.609∗(miles+yards/1760.0)=1.609∗(26+385/1760.0)=1.609∗(26+0.21875)=1.609∗26.21875=42.195

However, many compilers ignore this rule and parse it as e = ( ((a < d) ? (a++) : a) = d ), and then give an error because it is semantically invalid.

When parsing an expression, an operator which is listed on some row will be bound tighter (as if by parentheses) to its arguments than any operator that is listed on a row further below it. For example, the expression *p++ is parsed as *(p++), and not as (*p)++.

Operators that are in the same cell (there may be several rows of operators listed in a cell) are evaluated with the same precedence, in the given direction. For example, the expression a=b=c is parsed as a=(b=c), and not as (a=b)=c because of right-to-left associativity.

Notes

  • Precedence and associativity are independent from order of evaluation.
  • The standard itself doesn't specify precedence levels. They are derived from the grammar.
  • In C++, the conditional operator has the same precedence as assignment operators, and prefix ++ and -- and assignment operators don't have the restrictions about their operands.
  • Associativity specification is redundant for unary operators and is only shown for completeness: unary prefix operators always associate right-to-left (sizeof ++*p is sizeof(++(*p))) and unary postfix operators always associate left-to-right (a[1][2]++ is ((a[1])[2])++). Note that the associativity is meaningful for member access operators, even though they are grouped with unary postfix operators: a.b++ is parsed (a.b)++ and not a.(b++).

Putting it all together, the order of operations in the expression c = ++a + b++ + (d = a++); is as follows:

  • First, the compiler increments ++a, Now, a is 2.
  • Second, b++ uses the current value of b (which is 1) and then increments b by 1. Now, b is 2.
  • The reason why b is evaluated after ++a even though we have parentheses for (d = a++)
  • “+” associativity is left to right hence the expression is grouped like this
  • c = ((++a) + (b++)) + (d = a++)
  • Third, (d = a++) assigns the current value of a (which is 2) to d and then increments a by 1. Now, a is 3 and d is 2.
  • Finally, c is assigned the sum of the values: 2 (from ++a) + 1 (from b before it was incremented) + 2 (from d).
  • Therefore, the final value of c is 5.

Clarification on how the expression was evaluated like that >>

*****************************************************************************************************************

  • In the expression c = ++a + b++ + a++;, the order of evaluation is not determined by operator precedence but rather by the order of side effects and sequence points in C.
  • postfix ++ actually binds more tightly (has higher precedence) than prefix ++.
  • Precedence determines how the expression is parsed into a syntax tree, not the order in which each sub-expression is evaluated at runtime.
  • In C, the prefix increment ++a guarantees that a is incremented before its value is used in the expression. This is a sequence point, which means that all side effects (such as variable modifications) prior to that point must be completed before the next operation.
  • Therefore, ++a is evaluated first because it ensures that a is updated before any other parts of the expression use its modified value. This adherence to sequence points and side effects ensures consistent behaviour and avoids ambiguity in the evaluation of expressions in C.

In C programming, suffixing a numeric literal with L indicates that the literal should be treated as a long type. This is called a long integer constant.

The comma operator evaluates its first operand and discards the result, then evaluates its second operand and returns this value (and type). This allows multiple expressions to be evaluated in a single statement.

In your code, the expression (b = 7, c + d) is evaluated as follows:

b = 7 is evaluated first, and b is assigned the value 7. The result of this assignment is the value of b after the assignment, which is 7.

The comma operator then evaluates c + d. Since c and d are both initialised to 0, the result of c + d is 0.

The value of the entire expression (b = 7, c + d) is the result of the last evaluated expression, which is 0.

Therefore, the value of a after the assignment a = (b = 7, c + d); is 0.

The lecture focuses on the concept of call-by-value in function invocation, which is a fundamental aspect of how parameters are passed to functions in C. The key points covered include:

  • Call-by-Value: When a function is called, the values of the arguments are copied into the function's parameters. This means that any changes made to the parameters inside the function do not affect the original variables passed as arguments.
  • Example Function: The lecture provides an example of a function int compute_sum_to_n(int n) that calculates the sum of all integers from 1 to n. Inside the function, n is used in a loop to accumulate the sum, but n itself is not modified.
  • Parameter Conversion: If the argument passed to a function is of a different type than the parameter, the argument is converted to the parameter's type. This conversion happens before the function is called.
  • Return Statement: If a function has a return statement, the value of the expression in the return statement is returned to the calling environment. If there is no return statement, the function returns an implicit value, which depends on the function's return type.

C, returning an int (or any scalar type) by value is perfectly fine, even if that value was stored in a local (stack-allocated) variable. Here’s why:

  1. Value versus Storage
  • Your local variable sum lives on the stack while compute_sum_to_n is running.
  • When you do return sum;, the compiler generates code to copy the value of sum into the return value register (e.g., EAX on x86) or onto the caller’s stack slot (depending on the ABI).
  • Once the function returns, the caller reads that register/stack slot to get the returned integer.
  1. Lifetime and Scope
  • The storage for sum (the stack slot) is reclaimed when the function exits, but you’ve already copied its value out.
  • There’s no dangling-pointer issue, because you’re not returning an address into the local stack—just the integer itself.

The lecture covers the concept of scope rules in C programming, which are crucial for understanding how variables and functions are accessible within different parts of a program. Key points include:

  • Scope Rules: These rules determine the visibility and lifetime of variables and functions within a program. Variables and functions declared within a block are only accessible within that block and its nested blocks.
  • Lifetime of Variables: The lifetime of a variable begins when it is declared and ends when the block in which it is declared is exited. Variables declared within a block are not accessible outside that block.
  • Variable Shadowing: If a variable is declared within a nested block with the same name as a variable in an outer block, the inner variable shadows the outer variable. This means that references to the variable name within the nested block will refer to the inner variable, not the outer one.
  • Parallel Blocks: Variables declared in parallel blocks have separate lifetimes and are not accessible to each other.
  • Function Scope: Variables declared within a function are local to that function and are not accessible outside the function. The lifetime of these variables is from the point of declaration to the end of the function's execution.
